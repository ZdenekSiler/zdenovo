# zdenovo

Zdenek's personal blog and portfolio site — a FastAPI + SQLite backend serving
server-rendered HTML (Jinja2 + HTMX + Tailwind via CDN), with an AI-assisted
content pipeline that drafts blog posts using Claude for review before
publishing.

Built with [Claude Code](https://claude.com/claude-code) using a
design-then-implement-then-simplify workflow (see
[Claude Code Integration](#claude-code-integration)).

## Project Structure

```

├── backend/                  # FastAPI app
│   ├── main.py                # App, lifespan (DB init + scheduler), HTML page routes

├── nginx/, scripts/, Makefile, Dockerfile, docker-compose*.yml   # Deployment
└── .env.example                # Required environment variables
```

## Backend

- **FastAPI** app (`backend/main.py`) serves both server-rendered HTML pages and a
  JSON REST API.
- **SQLite** (`backend/blog.db`, gitignored) stores `posts` and `drafts`. Schema and
  seed data live in `backend/db.py`; on first run, three posts from
  `seed_posts.json` are inserted.
- **APScheduler** runs a daily job (02:00 UTC) that generates new draft posts via
  Claude (see [How Blog Posts Are Generated](#how-blog-posts-are-generated)).
- Run `uv run pytest` for the test suite (`backend/tests/`).

See [`.claude/rules/architecture.md`](.claude/rules/architecture.md) for module
layout, import/dependency rules, and where new code should go.

## Frontend

Plain HTML + Jinja2 templates (`frontend/templates/`), Tailwind CSS and HTMX loaded
via CDN — **no build step, no npm**. The sidebar nav in `base.html` uses HTMX
(`hx-get`, `hx-target="#main-content"`, `hx-push-url="true"`) to swap page content
without a full reload; `frontend/static/js/main.js` only highlights the active nav
link.

Pages: `/` (home), `/projects`, `/blog`, `/blog/{slug}` (single post), plus admin
views at `/admin/drafts` and `/admin/drafts/{id}`.

## API

JSON REST API under `/api/*`, self-documented via Swagger UI at `/docs`:

| Resource | Endpoints | Purpose |
|----------|-----------|---------|
| **Posts** | `GET/POST /api/posts`, `GET/PUT/DELETE /api/posts/{slug}` | CRUD for published posts |
| **Generation** | `GET /api/posts/briefs`, `POST /api/posts/generate`, `POST /api/posts/generate/{brief_id}` | On-demand AI post generation |
| **Drafts** | `GET /api/drafts`, `GET/DELETE /api/drafts/{id}`, `POST /api/drafts/generate`, `POST /api/drafts/{id}/approve` | Draft lifecycle |

Full endpoint reference with request/response shapes:
[`docs/architecture.md`](docs/architecture.md#api-endpoints).

## How Blog Posts Are Generated

Content can be written by hand (`POST /api/posts`) or generated by **Claude**
(`claude-sonnet-4-6` via the `anthropic` SDK, in `backend/routers/generate_api.py`):

1. A topic/brief (free-text `{description, tags}`, or a stored brief from
   `backend/data/post_briefs.json` / `daily_topics.json`) is rendered into a prompt.
2. Claude is called with a forced **`write_post`** tool — it must respond with a
   structured post (`title`, `summary`, `tags`, `content`, `image`), validated into
   a `PostOut` model.
3. **On-demand**: `POST /api/posts/generate` (or `/generate/{brief_id}`) returns the
   generated post directly — the caller decides whether to save it.
4. **Scheduled**: every day at 02:00 UTC (and via `POST /api/drafts/generate`),
   `generate_daily_drafts()` samples a few topics, generates a post for each, and
   inserts them into the `drafts` table with `status="pending"`.

### Reviewing & Publishing Drafts

- `/admin/drafts` lists pending drafts; `/admin/drafts/{id}` shows a full preview,
  rendered like a real post.
- `POST /api/drafts/{id}/approve` copies the draft into `posts` (marking it
  `status="approved"`) — this is the only way generated content reaches the public
  blog.
- `DELETE /api/drafts/{id}` discards a draft.

Generated content is **never** published automatically — every draft goes through
this preview/approve step.

## Getting Started

### Prerequisites

- Python 3.12+
- [uv](https://docs.astral.sh/uv/) — Python package manager
- Docker + Docker Compose (for `make dev`)
- An `ANTHROPIC_API_KEY` for AI generation — copy `.env.example` to `.env` and fill
  it in

### Local Development

```bash
make dev          # docker compose up --build -d → http://localhost:8080
make dev-logs     # tail logs
make test         # uv run pytest --cov (inside backend/)
make dev-down     # stop and remove dev containers
```

Or without Docker:

```bash
cd backend
uv sync
uv run uvicorn main:app --reload   # http://localhost:8000
uv run pytest
```

## Deployment

Single Hetzner VPS, Docker Compose, nginx + Let's Encrypt — fully automated via the
root `Makefile` (`make deploy-first`, `make deploy`, `make check`). See
[`docs/deployment.md`](docs/deployment.md) for the full guide.

## Claude Code Integration

This project is developed with Claude Code using a design-then-implement-then-simplify
workflow:

| Command | Description |
|---------|-------------|
| `/design <feature>` | Researches the codebase and writes a spec to `docs/specs/<feature>.md` |
| `/implement <feature>` | Implements the spec via parallel subagents |
| `/simplify` | Reviews changed code for duplication, over-engineering, and unused code |

Configuration:

- `CLAUDE.md` — loaded every session: architecture/workflow pointers, build/dev/deploy commands
- `.claude/rules/` — `architecture.md`, `code-style.md`, `testing.md`, `git.md`
- `.claude/skills/` — `design`, `implement`, `simplify`

See [`docs/workflows.md`](docs/workflows.md) for the full development workflow.
