# Spec: Deploy History

## Overview

Add a `/admin/deploys` page that shows a structured history of every production
deployment ŌĆö commit hash, timestamp, status, and duration ŌĆö written automatically
by the `make prod` Makefile target. Today the only deploy signal is a footer badge
showing the current commit and boot time; there is no record of past deploys, failed
deploys, or how long each one took.

---

## Current State

- `BUILD_COMMIT` is baked into the Docker image at build time via
  `BUILD_COMMIT=$$(git rev-parse --short HEAD)` in `Makefile:85` and written to
  `/app/BUILD_COMMIT` in `Dockerfile`. FastAPI reads it at startup and exposes it as
  a Jinja2 global in `base.html:207`.
- `deploy_time` is `datetime.now(timezone.utc)` at process start ŌĆö not a real deploy
  timestamp, resets on any container restart.
- No deploy event is ever persisted anywhere.
- The Makefile `prod` target already knows the commit hash before building, already
  runs `_post-deploy-check` after the container is healthy, and already has a clear
  success/failure branch ŌĆö ideal hook points for writing deploy records.

## Implementation Notes

### `backend/db.py` ŌĆö `deploys` table

Add after the existing `series` table block in `init_db()`:

```sql
CREATE TABLE IF NOT EXISTS deploys (
    id           TEXT PRIMARY KEY,
    commit_hash  TEXT NOT NULL,
    branch       TEXT NOT NULL DEFAULT 'main',
    deployed_at  TEXT NOT NULL,
    status       TEXT NOT NULL DEFAULT 'started',
    duration_s   INTEGER,
    triggered_by TEXT NOT NULL DEFAULT 'manual',
    notes        TEXT
)
```

Column notes:
- `id` ŌĆö UUID, generated by the caller (Makefile passes it in the POST body to allow a
  second PATCH call to update status/duration on the same record; simplest pattern:
  Makefile generates the UUID with `uuidgen` before the build, posts `status=started`,
  then posts the same UUID again with `status=success` or `status=failed` and
  `duration_s` after `_post-deploy-check` completes ŌĆö two POSTs, same ID). Wait ŌĆö
  simpler: make `POST /api/deploys` idempotent on `commit_hash + status` and just do
  two separate inserts; don't bother with update. Each deploy emits one `started` row
  and one `success`/`failed` row. The UI shows the most recent non-`started` row per
  deploy sequence, or just shows all rows.
  **Decision: simplest first.** One POST per deploy with `status` (`success` or `failed`)
  and `duration_s` sent in the same call, after the build/check completes. The `started`
  status is not stored ŌĆö it adds complexity for minimal value (knowing a deploy is
  currently in progress). If the deploy crashes mid-flight the record just won't exist.

- `duration_s` ŌĆö seconds from `make prod` invocation to `_post-deploy-check` passing.
  Computed in the Makefile with `START=$$(date +%s)` before the build.

- `triggered_by` ŌĆö `"makefile"` from automated Makefile calls, anything else for manual
  entries if added later.

- `notes` ŌĆö optional free text (reserved for future use, nullable).

No column migrations needed at v1 since the table is new.

### `secrets/deploy_token` ŌĆö shared secret

Add a new file `secrets/deploy_token` (gitignored via existing `secrets/*` pattern).
The Makefile reads it as `DEPLOY_TOKEN=$$(cat /opt/zdenovo/secrets/deploy_token)` and
passes it in a header: `curl -H "X-Deploy-Token: $$DEPLOY_TOKEN"`.

The FastAPI dependency in `deploys_api.py` reads it with `read_secret("deploy_token")`
(already-generic `config.py:read_secret()` supports any secret name). Use
`secrets.compare_digest()` for the comparison to prevent timing attacks ŌĆö never use `==`
for secret comparison. If the token file doesn't exist or the header is missing/wrong ŌåÆ
`HTTPException(401)`.

```python
import secrets as _secrets

def _verify_deploy_token(x_deploy_token: str = Header(...)) -> None:
    expected = read_secret("deploy_token") or ""
    if not _secrets.compare_digest(x_deploy_token, expected):
        raise HTTPException(status_code=401, detail="Invalid deploy token")
```

Generate the token with at least 32 random bytes:
`python3 -c "import secrets; print(secrets.token_hex(32))"`.

Add a placeholder in `.env.example`: `DEPLOY_TOKEN=changeme`.

### `backend/routers/deploys_api.py`

```
router = APIRouter(prefix="/api/deploys", tags=["deploys"])
```

**Schemas:**

```python
class DeployIn(BaseModel):
    commit_hash: str
    branch: str = "main"
    status: Literal["success", "failed"]
    duration_s: int | None = None
    triggered_by: str = "makefile"
    notes: str | None = None

class DeployOut(BaseModel):
    id: str
    commit_hash: str
    branch: str
    deployed_at: datetime
    status: str
    duration_s: int | None
    triggered_by: str
    notes: str | None
```

**Routes:**

`POST /api/deploys` ŌĆö auth via `X-Deploy-Token` header (not `require_admin` ŌĆö the
Makefile has no browser session). Insert row with `id=uuid4()`, `deployed_at=now()`.
Returns `DeployOut` (201).

`GET /api/deploys` ŌĆö requires `require_admin` (admin session, same as other GET
admin-data routes). Returns `list[DeployOut]`, newest first, no pagination (deploy
history grows slowly; 100-row cap is fine for v1: `LIMIT 100`).

### `backend/main.py` ŌĆö admin HTML route

```python
@app.get("/admin/deploys", response_class=HTMLResponse)
async def admin_deploys(request: Request, _: None = Depends(require_admin)):
    with get_conn() as conn:
        rows = conn.execute(
            "SELECT * FROM deploys ORDER BY deployed_at DESC LIMIT 50"
        ).fetchall()
    deploys = [deploy_row_to_dict(r) for r in rows]
    return templates.TemplateResponse(request, "admin_deploys.html", {"deploys": deploys})
```

`deployed_at` as `datetime`, pass the rest through).

Mount the router: `app.include_router(deploys_router)` following the existing pattern
in the router include block.

### `Makefile` ŌĆö `prod` target changes

```makefile
prod: _require-env _gen-nginx-conf _check-certs
    @echo "ŌåÆ Pre-deploy security check..."
    @bash scripts/pre-deploy-check.sh || (echo "ERROR: pre-deploy check failed ŌĆö aborting" && exit 1)
    @echo "ŌåÆ Backing up database before deploy..."
    @/opt/zdenovo/backup-db.sh || (echo "ERROR: backup failed ŌĆö aborting deploy" && exit 1)
    $(eval DEPLOY_START := $(shell date +%s))
    $(eval DEPLOY_COMMIT := $(shell git rev-parse --short HEAD))
    BUILD_COMMIT=$(DEPLOY_COMMIT) $(COMPOSE_PROD) up --build -d
    @$(MAKE) --no-print-directory _post-deploy-check \
        && DEPLOY_END=$$(date +%s) \
        && DURATION=$$((DEPLOY_END - $(DEPLOY_START))) \
        && DEPLOY_TOKEN=$$(cat /opt/zdenovo/secrets/deploy_token 2>/dev/null || echo "") \
        && curl -sf -X POST http://localhost:8000/api/deploys \
            -H "Content-Type: application/json" \
            -H "X-Deploy-Token: $$DEPLOY_TOKEN" \
            -d "{\"commit_hash\":\"$(DEPLOY_COMMIT)\",\"status\":\"success\",\"duration_s\":$$DURATION}" \
            && echo "  Ō£ō Deploy event recorded" \
            || echo "  ŌÜĀ Deploy event recording failed (non-fatal)" \
        || (DEPLOY_END=$$(date +%s) \
            && DURATION=$$((DEPLOY_END - $(DEPLOY_START))) \
            && DEPLOY_TOKEN=$$(cat /opt/zdenovo/secrets/deploy_token 2>/dev/null || echo "") \
            && curl -sf -X POST http://localhost:8000/api/deploys \
                -H "Content-Type: application/json" \
                -H "X-Deploy-Token: $$DEPLOY_TOKEN" \
                -d "{\"commit_hash\":\"$(DEPLOY_COMMIT)\",\"status\":\"failed\",\"duration_s\":$$DURATION}" \
                || true \
            && exit 1)
```

Note: `curl` failure is non-fatal on success path (so a missing `deploy_token` secret
doesn't break a working deploy). On the failure path, `curl` is also silenced
(`|| true`) so the Makefile exits 1 from `_post-deploy-check`, not from the curl.

**One-time server setup:** add `deploy_token` to the list of secrets pushed in
`deploy-first`:
```makefile
ssh $(SERVER_HOST) "echo '$(DEPLOY_TOKEN)' > $(DEPLOY_DIR)/secrets/deploy_token && chmod 600 $(DEPLOY_DIR)/secrets/deploy_token"
```
Also add `DEPLOY_TOKEN ?= $(shell cat secrets/deploy_token 2>/dev/null)` to the
Makefile's variable section so `make deploy-first` can read it locally.

## Risks & Trade-offs

1. **Makefile shell complexity.** The success/failure branching via `&&`/`||` in Make
   is fragile across different `make` versions. Mitigation: extract the record logic to
   `scripts/record-deploy.sh` and call that from the Makefile ŌĆö cleaner, testable,
   and easier to read than inline shell.

2. **`curl` to `localhost:8000` runs on the server.** This is fine since `make prod`
   runs on the server (`ssh zdenovo "... make prod"`). The container listens on 8000
   internally (confirmed by `_post-deploy-check`). No port needs to be opened to the
   public internet.

3. **deploy_token in plaintext on disk.** Same security posture as `admin_password` and
   `anthropic_api_key` ŌĆö files in `secrets/`, chmod 600, not in git. Acceptable for
   this trust level.

4. **History only starts from when this feature ships.** No backfill of past deploys.
   Expected and acceptable ŌĆö the table starts empty and grows forward.

5. **`notes` column reserved but not yet exposed in the UI.** Fine for v1; the column
   exists in the schema so it doesn't require a migration later.
